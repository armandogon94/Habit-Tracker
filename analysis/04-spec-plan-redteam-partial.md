## slice-00 — verdict: **needs-work**

**Summary:** slice-00-mockup.md is a well-organized, RED-first iOS scaffold+mockup plan that correctly front-loads visual sign-off and a signing dry-run. But it has serious latent problems grounded in ground-truth from the repo: (1) it is already partially executed — ios/project.yml, the xcodeproj, HabitTrackerApp.swift, RootView.swift, Info.plist and SmokeTest.swift already exist, yet Task 0.1 and the DoD treat the project as greenfield and the existing RootView contains NO TabView, directly contradicting 0.1's claim to produce one; (2) it bakes in an architecture that the very next slices tear out — slice-00 builds views with zero ViewModels (matching the 04-Finance-Tracker reference, which has none) but slice-03 introduces TodayViewModel/HabitDetailViewModel/etc. as NEW files and rebinds every view, so most slice-00 view code is thrown away; (3) it conflates domain models and wire DTOs — Task 0.4 puts Codable + snake_case decoding on the domain Models, but the 04 reference (and slices 02/03) deliberately keep Models (UI) and DTO (wire) as two separate type systems with service-layer translation, making slice-00's ModelCodableTests/snake_case design dead on arrival; (4) it adds zero localization while SPEC §7 mandates 'never hardcode Login' and Spanish-first — so the ~20 mockup screenshots that become the 'visual contract' will be hardcoded English, and slice-09 must retrofit String Catalog across all 12 screens, invalidating the baseline. Several acceptance criteria contradict the proven reference they say to 'port exactly' (ThemeStore.setTheme vs the real apply(); @MainActor vs the real plain @Observable). The signing dry-run (0.13) is mis-scoped and high-risk as the LAST task of a mock-only slice. Verdict: needs-work — the bones are good but the model/VM/localization seams and the already-started state will cause real rework if executed as written.

**Gaps:**

- State drift: the slice is already partially executed. `ios/project.yml`, `ios/HabitTracker.xcodeproj/`, `HabitTracker/HabitTrackerApp.swift`, `HabitTracker/RootView.swift`, `HabitTracker/Info.plist`, and `HabitTrackerTests/SmokeTest.swift` all exist on disk, but Task 0.1's 'Files to create' lists them as net-new and the DoD has no 'reconcile with existing scaffold' step. The existing RootView (read: a static VStack with 'Scaffold ready — slice 00 task 0.1') has NO TabView, yet 0.1 acceptance line 85 claims 0.1 produces a 'TabView with Today / Analytics / Settings'. An executor will either redo finished work or skip 0.1 and never build the TabView the later tasks assume exists.
- Domain-model vs DTO conflation. Task 0.4 (line 45/150) defines `Models/Models.swift` types as 'matching backend DTOs (Codable…)' and `ModelCodableTests` round-trips them with `.convertFromSnakeCase`. But the authoritative reference `04-Finance-Tracker/ios/FinanceTracker` keeps these as TWO layers on purpose: `Models/Models.swift` (UI structs, NOT Codable, use Color/UUID) vs `Core/Networking/DTO.swift` (Decodable wire types with explicit CodingKeys). Slices 02 and 03 re-create that split (`DTO.swift`, `DTO+Habits.swift`). So slice-00's Codable-on-domain-model and its snake_case test are building a type system that slice 03 will replace, and the 'matching backend DTOs' contract is untestable in slice 00 because no backend DTO schema is pinned yet (slice 01 defines `MobileTokenResponse`; habit DTOs aren't pinned until slice 03 Task 3.1).
- ViewModel architecture seam. slice-00 builds Today/HabitDetail/CreateEdit/Analytics with NO ViewModel files (consistent with 04, which has zero `*ViewModel.swift`). slice-03 then introduces `TodayViewModel`, `HabitDetailViewModel`, `CreateHabitViewModel`, `EditHabitViewModel`, `AnalyticsViewModel` as NEW files and 'binds' every view to them. The plan never states whether slice-00 views should be VM-ready or will be rewritten; as written, most slice-00 view internals (in-memory toggle state, mock fetch) are discarded in slice 03. Either slice-00 should introduce thin VMs now (so slice 03 swaps the data source, not the architecture) or the overview should explicitly mark slice-00 view bodies as throwaway. Neither is done.
- No localization infrastructure despite a hard SPEC boundary. SPEC §7 'Always do' says 'Localize every user-visible string via String Catalog; never hardcode "Login".' and §1 says Spanish-by-default. slice-00 creates zero `.xcstrings`, and every screen (Login, Register, Today hero 'X of Y done', Settings labels, FirstRun copy) hardcodes literals. The 04 reference also has no localization, so 'port from 04' propagates the violation. Consequence: the ~20 mockup screenshots in Task 0.12 — explicitly designated the 'visual regression baseline / visual contract' (DoD, §10) — will show hardcoded strings in one language, and slice-09 retrofitting String Catalog will change layout/line-wrapping and invalidate that baseline. Decide now: either set `SWIFT_EMIT_LOC_STRINGS` + a String Catalog and author strings Spanish-first in slice 00, or explicitly waive §7 for the mock and note the screenshots are non-final.
- Theme persistence collides with the SPEC token-storage rule's spirit and the localization toggle. ThemeStore persists in UserDefaults (fine; matches 04 key `FinanceTracker.selectedTheme`). But the FirstRunSheet 'theme pick' (Task 0.11) and Settings 'language toggle (placeholder)' (Task 0.10) both imply a persisted language choice that has no storage defined anywhere in slice 00, and SPEC promises a live Spanish↔English toggle. No `AppLanguageStore`/locale-override mechanism is specified, so the 'language toggle' is pure dead UI with no contract for slice 09 to build on.
- FirstRunSheet ordering is self-contradictory across slices. slice-00 Task 0.11 says FirstRunSheet shows 'on first launch (UserDefaults flag)' BEFORE the app (mock auth, no real sign-in). slice-02 Task 2.5 (line 38/140) says it shows 'only AFTER successful sign-in (or once, on app first launch)'. The UserDefaults flag key, and whether the gate is (first-launch) vs (first-post-auth), is never pinned. Whichever slice-00 picks may be wrong for slice 02, forcing rework of the exact onboarding flow the mockup is meant to validate with the user.
- RRULE timezone semantics are unspecified in the one place they're computed. Task 0.4's `RRULE.scheduledDates(in:)` and Task 0.6's 'today's scheduled habits per RRULE' must decide which calendar/timezone defines 'today' and weekday boundaries. CLAUDE.md and SPEC stress per-user IANA timezone and local-midnight resets, and slice-03 risk row flags 'Time-zone skew on completed_date'. slice-00 says nothing about Calendar/TimeZone, so the mock 'X of Y done today' and which habits appear today are timezone-undefined — and any choice made here silently becomes the contract the user signs off on.

## slice-01 — verdict: **needs-work**

**Summary:** Slice-01 is a competently-organized plan whose central premise — "every change is additive, no existing endpoint contract changes" — is contradicted by the actual code in backend/. Three load-bearing assumptions are false against ground truth: (1) the plan treats the DB password as "hardcoded" to be "removed," but config.py already loads everything via Pydantic BaseSettings (the password is a *default value*, and the real hardcoded credential lives in alembic.ini line 3 — the plan half-sees this); (2) Task 1.3 changes create_access_token from create_access_token(user_id) to create_access_token(user_id, role), which is a SIGNATURE CHANGE that breaks the three existing call sites in routers/auth.py (lines 23, 47, 72) — flatly contradicting the "additive only" objective; (3) the plan says "create tests/conftest.py" and a regression suite test_auth_web.py, but NO conftest, NO httpx test client, NO factories, and NO async-DB test harness exist today (the two existing test files are pure-unit, DB-free). Standing up an async Postgres + httpx.AsyncClient + factory-boy + fakeredis harness is a large, un-budgeted task hidden inside Tasks 1.2/1.4. The mobile refresh-token design (jti + Redis whitelist + rotation) is never reconciled with the existing cookie-based web refresh (get_refresh_token_payload reads a cookie; current refresh tokens have NO jti; /logout only deletes a cookie and revokes nothing) — so the codebase ends with two divergent refresh formats and a whitelist the web flow silently bypasses, which undermines the very revocation guarantee being sold. MobileTokenResponse is promised to carry expires_in and a user object plus a role claim, but expires_in semantics, the user shape, and adding role to UserResponse are never specified, and slice-02 already hard-depends on that unspecified shape. Verdict: needs-work — the skeleton and test-naming are good, but the plan must be corrected to (a) drop the false "additive" claim and explicitly own the security.py refactor + its call-site fan-out, (b) add an explicit "build the integration test harness" task as a hard prerequisite before 1.2/1.4, (c) fully specify MobileTokenResponse + reconcile web vs mobile refresh/whitelist/logout semantics, and (d) fix the dependency/secret-scanning/Redis-namespacing details below.

**Gaps:**

- FALSE PREMISE — 'additive, no contract changes' is contradicted by the plan's own Task 1.3. routers/auth.py currently calls create_access_token(str(user.id)) at lines 23, 47, and 72. Task 1.3 (line 108) requires create_access_token(user_id, role) — a signature change. Either it's a breaking change to a shared helper (and all three existing web call sites must be updated, which the 'Files' list for 1.3 omits — it lists security.py but not the web endpoints in auth.py that call it), or role must be optional with a default, which the plan never states. This contradiction sits at the heart of the slice objective (line 12) and is never resolved.
- SECRET-REMOVAL CLAIM IS HALF-WRONG. The DoD (line 228) and Task 1.1 (line 65) say 'remove hardcoded password from config.py.' But config.py line 5 is a *default value* on a Pydantic BaseSettings field that is already overridden by env (model_config env_file='../.env.local', line 14). The genuinely hardcoded plaintext credential is alembic.ini line 3 (sqlalchemy.url = ...:changeme_secure_password@...). The grep verify (`grep -r changeme_secure_password`, line 71) will FAIL because config.py keeps a placeholder default unless the author also edits the default — the plan never says whether to keep a safe default or delete it. The plan should target alembic.ini specifically and decide config.py's default policy explicitly.
- TEST HARNESS DOES NOT EXIST AND IS NOT BUDGETED. The plan says 'Create tests/conftest.py' (line 41) and writes router-level tests (test_auth_mobile, test_auth_web, test_ratelimit) that require an httpx.AsyncClient bound to the app with a real/throwaway async Postgres session, dependency overrides for get_db, and async-factory-boy. None of this exists: there is no conftest.py anywhere under tests/, no AsyncClient/TestClient usage, and the only two test files (test_streak_service.py, test_habit_rrule.py) are pure synchronous unit tests with zero DB. Building the async DB fixture (transactional rollback per test, get_db override, event-loop policy for asyncio_mode=auto) is itself a substantial task with real failure modes (asyncpg + pytest-asyncio loop scoping). It is silently folded into Tasks 1.2/1.4 instead of being its own prerequisite task. This is the single biggest schedule risk in the slice.
- MobileTokenResponse IS UNDER-SPECIFIED YET A HARD DEPENDENCY FOR SLICE 02. Acceptance (line 130) promises {access_token, refresh_token, expires_in, user}. But: (a) expires_in is undefined — seconds vs ms, access-token lifetime vs refresh? (b) the embedded 'user' object shape is never given; the existing UserResponse (schemas/auth.py) is {id, email, timezone} and does NOT include role — yet the whole slice adds role. (c) slice-02 line 26/55 says iOS DTOs 'match backend MobileTokenResponse shape exactly' and decode 'byte-perfect' from fixtures, so any ambiguity here becomes a cross-surface integration bug. The schema must be written out field-by-field (names, types, snake_case) in this slice, not left to implementation.
- WEB vs MOBILE REFRESH/LOGOUT/WHITELIST ARE NEVER RECONCILED. Today the web flow reads the refresh token from an httpOnly COOKIE (deps.get_refresh_token_payload, Cookie(default=None)), current refresh tokens contain NO jti (security.create_refresh_token encodes only sub/exp/type), and /logout merely deletes the cookie (revokes nothing server-side). The slice adds jti + Redis whitelist + rotation for *mobile only*. Result: two different refresh-token formats coexist, the whitelist is authoritative for mobile but completely bypassed by web, and 'immediate revocation' (a stated architectural goal in CLAUDE.md) still does not hold for web. The plan must state explicitly that web refresh is intentionally left cookie-only/unwhitelisted for this slice (and note the resulting asymmetry as accepted tech debt), or unify them. Right now it's an unacknowledged inconsistency.
- redis-py / fakeredis VERSION-MATCH AND ASYNC-API RISK UNDERSTATED. Task 1.2 builds the whitelist directly against redis.asyncio and tests against fakeredis.aioredis. The risk table (line 221) rates 'fakeredis API drift' as Low and defers real-Redis testing to slice 11. But fakeredis's async TTL/expiry semantics and SCAN behavior (needed for revoke_all_for_user, which implies scanning refresh:* or maintaining a per-user index) are exactly where fakeredis diverges. test_ttl_expiry (line 189) against fakeredis does not prove real expiry. revoke_all_for_user has no specified data model — scanning keys is O(N) and discouraged; a user->jti set index is needed but unspecified.
- NO PASSWORD-HASH MIGRATION / COMPAT NOTE despite touching auth. security.py uses bcrypt directly (import bcrypt, bcrypt.hashpw/checkpw) while pyproject declares passlib[bcrypt] (passlib is unused). This is pre-existing, but since this slice rewrites auth and adds mobile register/login that must hash/verify identically to the web path, the plan should at least note the bcrypt-vs-passlib discrepancy so an implementer doesn't 'helpfully' switch to passlib.CryptContext and silently break verification of existing demo@test.com hashes.

**Risks:**

- Editing routers/auth.py to add four mobile endpoints while the file also contains the working web endpoints is the highest-regression-risk activity in the slice, and the regression net (test_auth_web.py) does not exist yet and cannot exist until the async DB+client harness is built first. So during Tasks 1.4 the author is most likely to break web auth at exactly the moment they have no regression coverage. Sequencing makes this worse (see sequencing).
- slowapi behind Traefik: the plan's key func (line 220) plans X-Forwarded-For, but slowapi/limits defaults to remote_addr; behind Traefik every request shows the proxy IP unless XFF parsing is correct AND Traefik is trusted. Get this wrong and either (a) all users share one bucket (global lockout) or (b) clients spoof XFF to bypass limits. The test test_eleventh_login_returns_429 will pass locally (direct connection) and give false confidence; the prod behavior is untested until slice 11. This is a security control that is easy to ship broken-but-green.
- Rate limiting needs storage. slowapi's Limiter needs a storage_uri (in-memory by default, which does NOT work across multiple uvicorn workers/replicas and resets on reload). The plan never says whether the limiter uses Redis as its backing store. With Redis already wired for the whitelist, the natural choice is redis://...; if left in-memory, the 10/min guarantee is per-process and trivially defeated by the load balancer. Unspecified.
- Redis namespace collision: SPEC says Redis is 'shared from project 05' and config defaults to redis://localhost:6379/2 (DB 2 shared with this project's cache). The whitelist uses keys refresh:{jti}; slowapi (if Redis-backed) writes LIMITER/* keys; future caching (deferred) will also use DB 2. No key-prefix or DB-isolation discipline is stated, so a future FLUSHDB or a project-05 key pattern could wipe live refresh whitelists. The plan should mandate a project prefix (e.g. habits:refresh:{jti}).
- Redis-down failure mode is half-designed. Risk row (line 218) says 'fall back to no whitelist on dev only (env flag).' But if every login WRITES to the whitelist and every mobile refresh READS it, a Redis outage in prod means: logins may succeed (write fails silently?) but refresh fails closed → mass forced re-login, or fails open → revocation bypass. The fail-open/fail-closed policy per operation (login write, refresh read, logout delete) is never specified, and 'dev only' fallback means prod has no defined degraded behavior.
- Token rotation atomicity: Acceptance 1.4 (line 132) requires 'new pair; old jti revoked' and 'replaying old refresh → 401' (line 133). Without a single atomic Redis operation (e.g. check-and-delete via Lua/WATCH), two concurrent refreshes with the same old token can both pass the is_valid check before either revokes → two valid new families (session fixation/duplication, the exact risk row at line 217). The plan asserts the mitigation ('old revoked atomically') but specifies no mechanism, and the iOS side (slice-02 testConcurrent401sShareOneRefresh) only de-dupes on the *client*; a malicious/replaying client bypasses that.
- alembic env.py reads sqlalchemy.url from alembic.ini (config.get_main_option), NOT from Settings. Task 1.1 (line 65) says 'alembic.ini uses ${DATABASE_URL} interpolation OR env.py reads via Settings' — but alembic.ini does NOT do shell interpolation natively (configparser supports %()s, not ${}), so the only correct option is editing env.py to inject settings.DATABASE_URL. The plan offers a non-working option as if it were valid, inviting an implementer to write `${DATABASE_URL}` in alembic.ini and get a literal connection string. Must mandate the env.py route explicitly.
- PG enum migration footgun: Task 1.3 adds a Postgres ENUM type user_role. Postgres native ENUMs are notoriously painful (cannot easily add values in a transaction pre-PG12 nuance, downgrade requires DROP TYPE, autogenerate often does not emit CREATE TYPE / cleanup correctly). The plan says '<auto>' migration (line 40) implying autogenerate, but SQLAlchemy autogenerate frequently mishandles enum creation/drop. The downgrade path (drop column + drop type) is unspecified and is a common cause of broken `alembic downgrade`.
- DoD item 'manually promote one user to admin' via raw UPDATE (lines 111, 229) is an out-of-band manual step embedded as a completion criterion. It assumes demo@test.com exists in whatever DB the reviewer runs, is not reproducible in CI, and creates drift between environments. There is no seed/fixture for the demo user anywhere in the repo, so this step may simply fail for a fresh checkout.
- Pre-existing bcrypt 4.x + passlib incompatibility landmine: passlib 1.7.4 is in deps but unused; if an implementer touches hashing and routes through passlib, passlib 1.7.4 is known to crash reading bcrypt>=4 version (AttributeError on __about__). Because the slice runs `uv sync` and may pull a newer bcrypt, even the *existing* direct-bcrypt code is fine, but any drift toward passlib detonates. **Summary:** slice-03 reads as a clean, well-templated online-CRUD slice, but it was written against an imagined backend and an imagined iOS UI layer that do not match the repository. **Gaps:**

The plan treats streaks/`completed_today` as if they must be derived client-side or fetched separately, and Task 3.2 says the hero 'X of Y done today' and `isDoneToday` come 'from the VM' without noting the server already provides `completed_today`. The `HabitDTO` in §3 is never specified field-by-field, so the 'mirror exactly' acceptance criterion is untestable — there is no enumerated target. archived_at is nullable and present; description is nullable; rrule is always `FREQ=DAILY` — none of this is pinned.

But the analytics endpoint returns `weekly_counts: dict[str,int]` keyed by 3-letter weekday names ('Mon'..'Sun') aggregated over ALL history — not last-4-weeks, not per-date. The heatmap needs the `/calendar` (or `/logs`) endpoint; the weekly chart's '4 weeks' framing has no matching backend field. **Risks:**

## slice-04 — verdict: **risky**

**Summary:** Slice-04 aims to make streak computation RRULE-aware (weekday/weekly habits) plus add composite indexes, an ADR, and parametrized tests, sequenced as a deprecation-and-migration slice with a "snapshot old behavior" regression test. The intent is sound and the slice is well-formatted, but I verified its premises against the actual backend and found a fatal, plan-invalidating contradiction plus a cluster of false factual assumptions. CENTRAL BLOCKER: the slice is built to handle exactly two RRULEs — `FREQ=DAILY;BYDAY=MO,TU,WE,TH,FR` and `FREQ=WEEKLY;BYDAY=MO,WE,FR` — but `backend/app/schemas/habit.py::normalize_daily_rrule` ACTIVELY REJECTS both, and `backend/tests/schemas/test_habit_rrule.py` has pinned tests (`test_rejects_weekly_schedule`, `test_rejects_weekdays_only`, `test_rejects_interval_greater_than_one`) asserting that rejection is correct. No habit in the system can currently store a non-daily rrule (DB default is `FREQ=DAILY`, column is the only source). The slice never lists `schemas/habit.py` or that test file in its "Files to modify," so as written it would make the streak algorithm branch on data that cannot exist, while leaving the API contract that forbids the data untouched. The slice is not implementable in its current form; it is missing the entire "allow weekly/weekday RRULEs through validation + migrate/backfill" prerequisite, which is arguably a slice of its own. SECONDARY BLOCKERS, all verified: (1) `test_streak_service.py` already exists with 19 green tests that pin calendar-day semantics, yet the plan says "Create" it and Task 4.3 will silently invalidate those existing tests with no plan to reconcile them. (2) There is NO test database infrastructure — no `conftest.py`, no fixtures, no `factory_boy`/`async-factory-boy`/`aiosqlite`/`pytest-postgresql` in deps, and zero DB-backed tests exist — but Task 4.2/4.3 require a regression test against a "demo seed user" (habit A→8, habit B→0) and Task 4.4 requires `EXPLAIN ANALYZE` against a live `habits_db`; none of this scaffolding exists or is budgeted. (3) No seed/demo data exists anywhere in the repo, so the asserted streak values are fabricated. (4) `docs/adr/` does not exist and `app/routers/analytics.py` does not exist (referenced by DoD §10). (5) Signature mismatch: `compute_current_streak(db, habit_id, today)` takes a UUID today, but Task 4.3 writes `compute_current_streak(db, habit, today)` and simultaneously claims habits.py has "no signature change" — internally contradictory; every call site (router + habit_service) passes `habit.id`. (6) The chosen anchor `habit.created_at` is a TIMESTAMPTZ datetime used as the basis for DATE-level RRULE expansion; anchor/timezone semantics are undefined and interact badly with the DST edge cases the plan claims to test. Verdict: risky — the slice rests on a false premise (the data it computes on is forbidden by the very codebase it lives in) and on test/seed infrastructure that does not exist.

**Gaps:**

- FATAL: The rrule validator in backend/app/schemas/habit.py rejects FREQ=DAILY;BYDAY=... and FREQ=WEEKLY;BYDAY=... — the exact two schedules slice-04 is built to support. backend/tests/schemas/test_habit_rrule.py pins this rejection (test_rejects_weekly_schedule, test_rejects_weekdays_only). The slice does not list schemas/habit.py or that test file in 'Files to create/modify'. Result: the RRULE-aware algorithm would branch on data that cannot enter the system, and the API still 422s any attempt to create such a habit. The 'allow non-daily RRULEs through validation' work is entirely missing and is a precondition for every other task in the slice.
- No test-DB infrastructure exists. `find backend -iname conftest*` returns nothing; there are zero DB-backed tests; factory_boy/async-factory-boy/aiosqlite/pytest-postgresql/testcontainers are NOT in pyproject.toml or uv.lock. Yet Task 4.2 ('runs against demo seed user'), Task 4.3 (compute_*(db, habit, today) is async and needs an AsyncSession), and the regression test all require a working async test database + fixtures + factories. This foundational scaffolding is neither present nor a task in the slice. It alone is a multi-task effort (conftest, engine/session override, schema create/drop, factories) and is unbudgeted.
- No seed/demo data exists anywhere in the repo (grep for seed|demo across backend/app, backend/tests, backend/alembic finds only an unrelated code comment). Task 4.2's concrete assertions ('habit A → 8, habit B → 0') and Task 4.3's 'update expected values' are therefore fabricated against data that does not exist. There is no seed script, no migration data, no fixture set to snapshot.
- Task 4.3 acceptance writes `compute_current_streak(db, habit, today)` but the real signature is `compute_current_streak(db, habit_id: UUID, today)` and every caller (routers/habits.py x3, services/habit_service.py x2) passes `habit.id`. The plan also asserts routers/habits.py needs 'no signature change; just verify behavior' — contradictory. Either the streak functions change signature (touching 5 call sites the plan doesn't enumerate) or they must separately load the habit's rrule by id (extra query the 'avoid N round-trips' goal forbids). This ambiguity is unresolved.
- Anchor semantics undefined. `expected_dates(rrule, start, end, anchor)` and `is_scheduled(rrule, d, anchor)` take an `anchor`, and Task 4.3 passes `habit.created_at`, which is a TIMESTAMPTZ (datetime with tz), not a date. RRULE expansion over DATE values needs a date anchor in the user's timezone; created_at is a UTC instant. For FREQ=WEEKLY this changes which weekday the week is anchored to and interacts directly with the DST cases the plan claims to cover. The plan never specifies how the datetime is reduced to an anchor date, in which timezone, or what BYDAY does to the anchor (BYDAY makes the anchor's weekday irrelevant for weekly-with-BYDAY, but not for plain weekly).
- docs/adr/ directory does not exist (Task 4.5 assumes docs/adr/0001-...). Minor but the 'linked from CLAUDE.md and SPEC.md §11' step edits SPEC.md, which the spec header marks as the approved contract — editing it to add an ADR link is an unflagged scope creep into a frozen document.
- DoD §10 claims 'calendar/analytics endpoints scale better,' but app/routers/analytics.py does not exist (only auth.py and habits.py). Analytics lives inside habit_service.get_analytics, which runs its OWN full-table scans (select all completed_date, Counter over weekday) that the (habit_id, completed_date) index does NOT optimize (no ORDER BY ... LIMIT). The performance claim is partly unfounded and the analytics N+1 in list_habits (per-habit current+longest+completed_today = 3 queries x N habits) is the real perf problem, which this slice ignores.

**Risks:**

- Silent test-suite erosion: Task 4.3 rewrites streak_service.py whose pure helpers current_streak_from_dates/longest_streak_from_dates are directly imported and asserted by the existing 19-test test_streak_service.py. If the rewrite changes those signatures (e.g., to take rrule/anchor), all 19 existing tests break or get overwritten. The plan treats test_streak_service.py as a new file and never mentions the 19 existing assertions, so an implementer will likely clobber a green, behavior-pinning test file — exactly the regression-safety the slice claims to provide via 4.2.
- The 'snapshot old behavior then let it fail' pattern (Task 4.2 → 4.3) is the WRONG tool here and actively harmful: there is no production data with non-daily rrules to protect (all habits are FREQ=DAILY), so for daily habits the new algorithm MUST produce identical results — there is nothing to snapshot-and-break. The regression file will either (a) be trivially identical pre/post for daily data, providing zero audit value, or (b) be built on fabricated weekly seed data that cannot be created through the API. The deprecation-and-migration framing is misapplied.
- EXPLAIN ANALYZE verification (Task 4.4) requires a populated `habits_db` reachable via `psql habits_db -c ...`. In this dev setup the DB runs in Docker (per CLAUDE.md ports), is likely empty, and `psql` may not be on PATH. An empty/tiny table makes Postgres choose a seq scan regardless of indexes, so the acceptance criterion 'shows index scan, not seq scan' will FAIL for correct work — a false-negative gate. No data-volume precondition or seeding step is specified.
- Composite-index redundancy / over-engineering: ix_habit_logs_habit_id and ix_habit_logs_completed_date already exist. Postgres can scan a (habit_id, completed_date) btree backward to satisfy ORDER BY completed_date DESC, so the separate (habit_id, completed_date DESC) index is largely redundant and just doubles write cost and bloat. The plan adds BOTH directions without measuring; at the stated 'small data' scale neither index will even be chosen. This is premature optimization the plan elsewhere claims to avoid.
- Cross-slice contract risk: Slice 05 Task 5.5 ports schedule.py + streak_service.py to Swift (StreakComputer) and asserts 'matches backend output for all test cases.' Slice 04 freezes neither the function set, the anchor semantics, nor the rrule grammar. Any anchor/timezone ambiguity or rrule-grammar decision made loosely in 04 silently becomes a correctness bug duplicated in Swift, and the two implementations will diverge on DST/anchor edge cases that the parametrized matrix doesn't actually nail down (the 'expected' values in the matrix are unspecified — '(rrule, days, expected)' with no values given).
- dateutil.rrule expansion over large date ranges (e.g., longest_streak over a multi-year history) materializes every scheduled occurrence in Python. For FREQ=DAILY that's 365 * years dates per habit per request, recomputed on every list_habits call for every habit (no caching despite the architecture note saying 'cache in Redis if perf becomes an issue', and Redis is 'NOT YET WIRED' per SPEC §2). The 'avoid N round-trips to DB' note addresses query count but not the O(occurrences) Python expansion now added on every request.
- Editing SPEC.md (Task 4.5) and CLAUDE.md to add ADR links: SPEC.md is the approved/frozen contract and CLAUDE.md is committed project memory. SPEC §7 'Ask first' covers schema changes but the plan modifies the spec document itself without flagging it — a governance smell that can cause merge friction and spec drift.
